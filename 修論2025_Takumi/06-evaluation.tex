\section{Evaluation}\label{sec:evaluation}

This chapter presents the evaluation of the BDEC credential verification relation $\mathcal{R}_{\textsf{ShowVer}}$ as implemented in the Noir + Aurora static circuit backend. We proceed in four stages. First, we establish the evaluation methodology and scope (\S\ref{sec:methodology}). Second, we present the general performance landscape of zkVMs versus circuit compilers using published benchmark data, establishing the broader context in which our measurements sit (\S\ref{sec:landscape}). Third, we present comprehensive measured results for the Noir + Aurora backend, including constraint decomposition, scaling behavior, and end-to-end performance (\S\ref{sec:loquat-costs}). Fourth, we quantify update churn through concrete deployment scenarios, drawing on the structural properties of both backends (\S\ref{sec:churn-eval}). We conclude with a discussion of what the measured static-path baseline implies for the zkVM path (\S\ref{sec:zkvm-outlook}).


\subsection{Evaluation Methodology}
\label{sec:methodology}

\subsubsection{Evaluation Dimensions}

We evaluate both backends along two co-equal dimensions introduced in Chapter~\ref{sec:prelim}:
\begin{itemize}
\item \textbf{Dimension D2 (Runtime Cost):} Prover time $T_P$, verifier time $T_V$, proof size $|\pi|$, and constraint count $N_C$.
\item \textbf{Dimension D1 (Update Churn):} Recompilation time, re-setup time, update bandwidth, and coordination overhead for each update event type (Definition~\ref{def:update-events}).
\end{itemize}

\subsubsection{Data Sources and Methodology}

Our evaluation draws on three layers of data, each with different levels of directness.

\paragraph{Layer 1: Published Benchmarks (Directly Measured).}
We reference two independent benchmark suites that provide measured prover times, proof sizes, and cycle counts across multiple frameworks:
\begin{itemize}
\item The \emph{babybear-labs benchmark}~\cite{RISCZeroBenchmarks2024}, conducted on AlmaLinux 8.10 with 16 cores and 32~GB RAM, evaluating Fibonacci, SHA-256, and Poseidon hash across RISC Zero, SP1, Jolt, Circom, and Halo2.
\item The \emph{ZKP Frameworks Survey}~\cite{Gassmann2025}, conducted on an AMD Ryzen 3990X with 128~GB RAM, evaluating matrix multiplication and SHA-256 across Gnark, Arkworks, SnarkJS, Noir, RISC Zero, MidenVM, and others.
\end{itemize}

\paragraph{Layer 2: Loquat-Aurora Benchmarks (Author-Reported).}
The Loquat paper~\cite{Loquat2024} reports aggregate signature benchmarks using the Aurora and Fractal implementations in \emph{libiop}, conducted on an Ubuntu VM with 28~GB RAM hosted on an Intel Core i7-12700KF at 3.6~GHz.

\paragraph{Layer 3: Our Implementation Results.}
We implement the BDEC.ShowVer verification logic for the Noir + Aurora backend (as described in Chapter~\ref{sec:system}) and report measured prover time, verifier time, proof size, and constraint counts. For the RISC Zero backend, we derive theoretical cycle count predictions from the analytical model in Chapter~\ref{sec:Theoretical Bottleneck}, validated against published RISC Zero benchmark data from Layer~1.

\paragraph{Experimental Setup.}
All Aurora measurements were conducted on an Apple M2 MacBook Air (8-core CPU, 8~GB unified memory) running macOS 15.4 (Darwin 25.4.0), compiled with \texttt{rustc} 1.91.1 in \texttt{--release} mode. Each configuration was executed for $n = 10$ measured runs preceded by 2 warmup runs (discarded). We report means and standard deviations across the 10 measured runs. Results are recorded in JSONL format via the \texttt{bench\_runner} harness (Appendix~\ref{app:repro}).

\paragraph{Scope Limitation.}
Our RISC Zero implementation was not exercised in time for this evaluation due to a verification bug in the Legendre PRF witness generation path. Consequently, all RISC Zero figures in this chapter are \emph{theoretical predictions} derived from the cost model of Chapter~\ref{sec:Theoretical Bottleneck}, clearly marked as such. We validate these predictions against published external benchmarks where possible.

\subsubsection{Key Observations from the Landscape}
\label{sec:landscape-observations}

\paragraph{Observation 1: The performance gap is operation-dependent.}
For algebraic operations (Poseidon, Fibonacci), circuit compilers outperform RISC Zero by $3.6$--$32.5\times$. For bitwise operations (SHA-256 over large inputs), RISC Zero \emph{outperforms} Circom by $18.4\times$ due to the SHA-256 precompile. Blanket ``zkVMs are slower'' claims are structurally misleading.

\paragraph{Observation 2: The backend choice within circuit compilers matters enormously.}
For SHA-256, Gnark (Groth16) is $28.1\times$ faster than RISC Zero, while Circom is $18.4\times$ \emph{slower}---a $512\times$ spread across circuit compiler implementations for the same operation.

\paragraph{Observation 3: Cycle counts reveal trace overhead.}
For Poseidon, Circom produces 4,184 constraints while RISC Zero requires 524,288 cycles---a $125\times$ trace amplification, exceeding the per-operation $\Theta(k^2)$ bound and indicating substantial fixed costs (initialization, paging, ISA emulation) beyond the arithmetic mismatch.

\paragraph{Policy and Revocation Overhead.}
Policy enforcement adds negligible constraint overhead. Adding a GPA $\geq$ 3.0 range check increases the constraint count by 291 ($+0.029\%$), and adding a degree-set membership check adds a further 7 constraints ($+0.001\%$). Increasing the revocation tree depth from 10 to 20 adds 3{,}120 constraints ($+0.31\%$), corresponding to 10 additional Merkle path levels at $\sim$312 constraints per level.

The Loquat paper~\cite{Loquat2024} reports aggregate signature benchmarks with Aurora: $n = 4$ signatures in 14~s, $n = 8$ in 33~s, $n = 16$ in 67~s, $n = 32$ in 160~s, $n = 64$ in 553~s, measured on an Intel i7-12700KF (3.6~GHz, 28~GB RAM). The constraint count matches the prediction from Equation~\ref{eq:linear-scaling}: $501{,}665 + 247{,}429 \cdot 1 + 298 = 749{,}392$, confirming the additivity of policy constraints.


\subsubsection{Implications of the Static-Path Baseline for the zkVM Path}
\label{sec:comparison}

The Aurora measurements establish a concrete baseline against which the zkVM path can eventually be compared. Several features of the measured data are directly relevant to understanding what the zkVM path will face.

\paragraph{The Base Overhead Dilutes the Mismatch Effect.}
The most important finding for the eventual cross-backend comparison is that 64.6\% of the Aurora circuit cost comes from credential binding and infrastructure overhead, not from the Loquat signature verifications themselves. This base overhead includes commitment schemes, selective disclosure infrastructure, and IOP bookkeeping that are \emph{not} dominated by Griffin hash. Since the arithmetic mismatch theory (Chapter~\ref{sec:Theoretical Bottleneck}) predicts overhead primarily on field-mismatched operations like Griffin, the dilution by base costs means that the realized agility tax on the full ShowVer relation will likely be lower than the per-operation theoretical bound of $7$--$30\times$. The precise magnitude, however, can only be determined through actual zkVM measurements.

\paragraph{Aurora's Proof Size and Verification Time Are Weak Points.}
The 32.0~MB IOP transcript size and 79.7~s verification time are notable weaknesses of the Aurora backend. (Note that the 32.0~MB figure is the full IOP-model transcript; under the BCS transformation, the succinct proof would be $O(\log^2 n) \approx 130$--$170$~KB, though our implementation measures the pre-compilation transcript.) RISC Zero's STARK proofs are architecturally bounded at $O(\log T)$ verifier complexity and typically produce proofs in the hundreds of kilobytes. Once zkVM measurements are available, the comparison will likely reveal a multi-dimensional trade-off rather than a unidirectional penalty: the static path may win on prover time while losing substantially on proof size and verification time.

\paragraph{The Scaling Linearity Enables Projection.}
The perfect linearity of the scaling law (Equation~\ref{eq:linear-scaling}, $R^2 = 1.000$) means that Aurora costs for higher credential counts ($k > 2$) can be projected with high confidence. This same linearity will serve as a prediction anchor once zkVM measurements are available: if the zkVM also scales linearly (as expected from the trace model), the agility tax ratio will converge as $k$ increases, since the marginal cost per credential is dominated by Griffin operations where the mismatch effect is strongest.


\subsubsection{Validation of the Precompile Paradox}
\label{sec:precompile-validation}

The benchmark data provides empirical evidence supporting the Precompile Paradox (Proposition~\ref{prop:precompile-paradox}).

In the babybear-labs benchmark, RISC Zero proves SHA-256 (1~KB) in 2.50~s while Circom requires 46.07~s---RISC Zero is $18.4\times$ faster due to the SHA-256 precompile. Conversely, for Poseidon (algebraic, no precompile), Circom is $4.6\times$ faster.

Griffin hash occupies the same structural position as Poseidon: an algebraically-structured hash operating natively over a large prime field, with no RISC Zero precompile. Our component analysis confirms that Griffin verification accounts for the entirety of the per-credential marginal cost in R1CS (172{,}765 constraints for a single Loquat verification). If the Loquat protocol were redesigned to use SHA-256 instead of Griffin, the RISC Zero cost would drop dramatically (SHA-256 precompile: 68 cycles per block), but the R1CS cost would explode ($\sim$826{,}000 constraints for $\sim$28 SHA-256 compression blocks~\cite{Poseidon2021}, i.e., $\sim$29{,}000 per single block~\cite{circomlib}, versus $\sim$304 per Griffin permutation in our R1CS encoding)---confirming the paradox that the optimal hash choice is opposite for the two backends.


\subsection{Update Churn Analysis}
\label{sec:churn-eval}

We evaluate Update Churn ($D_1$) through three concrete scenarios from Definition~\ref{def:update-events}. Unlike the runtime cost analysis, churn is assessed qualitatively based on the structural properties of each backend, as the relevant operations (circuit rewriting, key regeneration, network coordination) span timescales from seconds to months and involve human effort that cannot be meaningfully benchmarked.

\subsubsection{Scenario 1: Policy Update---Adding a GPA Threshold}

\paragraph{Noir + Aurora.}
Adding a GPA $\geq$ 3.0 check requires modifying the Noir source (minutes of developer effort), recompiling to R1CS (seconds--minutes), regenerating Aurora indexer keys ($\sim$87~s on our hardware), and distributing the new $vk_{\textsf{ZK}}'$ to all verifiers. Our measurement confirms that the constraint overhead is negligible (+291 constraints), but the key regeneration and verifier coordination dominate. For blockchain deployments, propagating a new verification key requires a consensus upgrade. \textbf{Total: hours to days.}

\paragraph{RISC Zero.}
Adding \texttt{assert!(credential.gpa >= 30);} to the Rust guest source, recompiling to RISC-V (seconds), and propagating the new 32-byte imageID. No key regeneration, no consensus upgrade---the verification infrastructure accepts any valid receipt whose imageID matches the whitelist. \textbf{Total: minutes.}

\subsubsection{Scenario 2: Algorithm Rotation---Loquat to ML-DSA}

\paragraph{Noir + Aurora.}
Complete circuit rewrite. ML-DSA verification involves polynomial ring operations over $\mathbb{Z}_q[x]/(x^n + 1)$, which are fundamentally incompatible with the Griffin-optimized R1CS structure. The operating field $\mathbb{F}_p$ ($p = 2^{127} - 1$) may need to change to accommodate ML-DSA's native modulus. All cryptographic parameters, keys, and verifier infrastructure must be regenerated and redeployed. \textbf{Total: weeks to months.}

\paragraph{RISC Zero.}
Library swap: replace the Loquat crate with \texttt{pqcrypto-dilithium}. ML-DSA's 32-bit NTT operations run naturally on RISC-V's rv32im ISA. The same AIR infrastructure, the same proof format, and the same verification pipeline apply. Only the imageID changes. \textbf{Total: hours to days.}

\subsubsection{Scenario 3: Schema Extension---Adding Honors Distinction}

\paragraph{Noir + Aurora.}
New witness wire, new hashing constraints, invalidated keys, verifier redeployment. \textbf{Total: days to weeks.}

\paragraph{RISC Zero.}
New struct field, updated hash computation, conditional disclosure in standard Rust. \textbf{Total: hours.}

This difference is structural: Aurora's verification key encodes the full R1CS structure, while RISC Zero's imageID is a hash of the guest binary, decoupling verification infrastructure from circuit parameters.


\subsection{Outlook: What the Baseline Implies for the zkVM Path}
\label{sec:zkvm-outlook}

The measured Aurora baseline, combined with the theoretical framework of Chapter~\ref{sec:Theoretical Bottleneck} and the published benchmark landscape (\S\ref{sec:landscape}), allows us to reason about the eventual zkVM comparison without fabricating specific numbers. The arithmetic mismatch theory establishes a structural $\Omega(k^2)$ overhead for non-native field operations, and the Precompile Paradox (validated empirically for Poseidon vs. SHA-256 in \S\ref{sec:precompile-validation}) confirms that Griffin hash---which dominates each Loquat verification---will be the primary cost driver in the zkVM path.

At the same time, the 64.6\% base overhead discovered in our component analysis means that the \emph{realized} agility tax on the full ShowVer relation will be substantially lower than the per-operation bound. The precise magnitude is an empirical question that can only be resolved through complete zkVM receipt measurements, identified as the highest-priority future work in Chapter~\ref{sec:conclusion}.

Importantly, the update churn analysis (\S\ref{sec:churn-eval}) does \emph{not} depend on zkVM runtime measurements---it follows from the structural properties of the two architectures. The $10^6\times$ bandwidth asymmetry (32~MB verification key vs. 32-byte imageID) and the decoupling of verification logic from cryptographic parameters are architectural facts, not performance predictions.

\subsection{Chapter Summary}

\textbf{Finding 1: The credential infrastructure dominates Aurora circuit cost.} Contrary to the expectation that Griffin hash would dominate the full circuit at $> 90\%$, our component analysis reveals that credential binding and base overhead account for 64.6\% of the total R1CS constraints, with the two Loquat signature verifications contributing only 34.7\%. Griffin hash dominates \emph{within} each Loquat verification (consistent with the Loquat authors' 93.5\% figure for Loquat-80), but the full credential system introduces substantial overhead from commitment schemes, selective disclosure infrastructure, and IOP bookkeeping. The scaling is perfectly linear at 247{,}429 constraints per additional credential.

\textbf{Finding 2: Aurora's IOP transcript size and verification time are significant deployment constraints.} The 32.0~MB IOP transcript size (full oracle evaluation; the BCS-compiled succinct proof would be $O(\log^2 n) \approx 130$--$170$~KB) and 79.7~s verification time represent the primary practical limitations of the static circuit backend as measured. These metrics reflect the IOP-model implementation and do not improve with circuit optimization. They motivate the zkVM alternative, where STARK-based proofs are asymptotically smaller and faster to verify.

\textbf{Finding 3: Update churn is categorically different between architectures.} Policy updates take minutes (RISC Zero) versus hours-to-days (Aurora). Algorithm rotations take hours-to-days (RISC Zero) versus weeks-to-months (Aurora). Update bandwidth differs by $10^6\times$ (32 bytes vs.\ 32~MB). This structural difference is independent of runtime performance measurements and holds regardless of the eventual zkVM prover time.

\textbf{Scope of Evidence.} All runtime measurements in this chapter are from the Noir + Aurora backend. The zkVM implementation (RISC Zero) is designed and partially implemented as described in Chapter~\ref{sec:system}, but complete receipt-side benchmarks were not obtained within the timeline of this work. The theoretical framework of Chapter~\ref{sec:Theoretical Bottleneck} provides directional expectations for the eventual comparison, but we deliberately refrain from presenting model-derived numbers as substitutes for measurements.
